# Feature Specification: View Agents — Repository Views & Parallel Local-Model Sub-Agents

**Feature Branch**: `001-view-agents`
**Created**: 2026-04-09
**Status**: Draft
**Input**: User description: "Create a view-agents system: repository views for progressive disclosure, mental models stored hierarchically in views, and a Python/Ollama CLI sub-agent that the discovery agent can parallelize work across views using local models."

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Generate Repository Views (Priority: P1)

A developer (or automated pipeline) points the view generation tool at a repository. The agent analyzes the repository structure and generates **per-view Python scripts** that produce a set of **views** — each view is a directory containing symbolic links (or copies) to a coherent subset of the repository files, organized for progressive disclosure. Views are stored under a `views/` directory at the repository root. Each view includes its own `regen.py` Python script (generated by the agent, tailored to that specific view) so the view can be refreshed independently when the repository changes. These scripts are project-specific artifacts — not generic tooling — because the optimal file groupings depend on the repository's structure, conventions, and domain. Having one script per view (rather than one global script) enables granular staleness detection: when a view's `regen.py` changes, only that view's mental models and the root mental model sections referencing that view are flagged — not every view in the repository.

**Why this priority**: Views are the foundational artifact everything else depends on. Without views, there is nothing for local models to reason over and no structure for mental models.

**Independent Test**: Point the agent at a sample repository. Verify that a `views/` directory is created containing at least one named view subdirectory, that each view contains symbolic links to real files and its own `regen.py` script, and that running a view's `regen.py` recreates that specific view when re-run.

**Cucumber Test Tag**: `@view-generation`

**Acceptance Scenarios**:

1. **Given** a repository path, **When** the agent generates view scripts, **Then** a `views/` directory is created at the repository root containing one or more named view subdirectories, each with symbolic links into the source tree and its own `regen.py` Python script.
2. **Given** a previously generated view, **When** the repository contents change and that view's `regen.py` is re-run, **Then** the view is updated to reflect the current repository state (new/moved/deleted files) independently of other views.
3. **Given** a repository with no recognizable structure, **When** the agent generates view scripts, **Then** it produces a single flat "all-files" view as a fallback.
4. **Given** a completed ticket that modified the repository's file structure, **When** the individual `regen.py` scripts are re-run, **Then** the views are updated and mental models for views whose file set changed are flagged stale.

### User Story 2 — Mental Models with Chain of Custody (Priority: P1)

Each view (and the views root) contains a `mental-models/` subdirectory. Mental models are markdown documents (wiki-style) that describe the purpose, architecture, patterns, and key decisions of the code visible in that view. Mental models replace ad-hoc documentation and agent memory files with a structured, hierarchical knowledge base that evolves with the code.

Critically, mental models only reference a **curated subset** of inputs. For a view-level mental model, those inputs are the source files the LLM judged important enough to inform the mental model. A view may contain hundreds of files, but the mental model might only reference 10-20 of them. For the root mental model, those inputs are child view mental-model sections. This is by design: by forcing the LLM to choose which files or child sections matter, the system creates an emergent curation of what is architecturally significant. The chain of custody only tracks these selected inputs, not every file in the view.

Every mental model tracks its **chain of custody** via **immutable metadata files**. Each time the mental model is updated, a new metadata file is created (never mutated) containing diff entries. Each diff entry covers one mental-model section hunk, identified by a unique section path (the full markdown heading hierarchy), and includes the curated inputs that support that hunk plus a pointer to the previous metadata file whose diff entry this one supersedes. For view-level mental models, the curated inputs are source files from that view. For the root mental model, the curated inputs are child view mental-model sections. The head metadata file references live repository paths or child head metadata IDs; older metadata files are fully self-contained snapshots. This forms a flat, hash-map-like structure of immutable files — walking backward is just following pointers, with no log parsing or git operations needed.

When any of these selected inputs changes, the mental model sections derived from that input are marked invalid. This invalidation propagates hierarchically: a view-level mental model's diff entries reference selected source files; the root-level mental model's diff entries reference specific sections of child view mental models. If a selected source file changes, the view mental-model diff entries referencing it are invalidated. If a root diff entry references that now-stale child section, the root diff entry is invalidated as well. The root has no `regen.py`; root invalidation happens only through child mental-model references.

Critically, **the LLM never interacts with the metadata files directly**. A set of helper scripts provides a clean interface for viewing and updating mental models:

- **`view-model show`** — renders the current mental model with staleness annotations, resolving all metadata internally. The LLM sees the mental model content with markers on stale sections, plus a list of the curated references that support it. For view-level models these are source file paths; for the root model these are child view mental-model references. No metadata file paths are exposed.
- **`view-model files`** — lists the curated references for the mental model (or a specific section), with their current status (current/stale). For view-level models, this returns source file paths. For the root model, this returns child view mental-model references.
- **`view-model update`** — accepts updated mental model content and a list of section references. For view-level mental models, references are source file paths plus optional source spans (`start_line` + `line_count`). For the root mental model, references are child view names plus child section paths. The script handles all metadata mechanics internally: creates the new immutable metadata file, copies snapshots, links diff entries to the previous metadata file, updates the head pointer. The LLM just provides content and references.
- **`view-model add-ref`** — adds a curated reference to an existing mental model section without changing the content. In a view mental model, that reference is a source file. In the root mental model, that reference is a child view mental-model section.
- **`view-model status`** — reports which sections are current vs. stale, without rendering the full content.

These scripts are the same reusable tool referenced in FR-009f — they abstract the immutable metadata chain so that all consumers (the view-agent CLI, the search tool, developers) interact through a consistent interface.

**Why this priority**: Promoted to P1 because mental models that go stale silently are worse than no mental models at all. The chain of custody is essential for the MVP — agents must know which parts of a mental model they can trust and which need regeneration. The curated-subset approach also ensures the chain of custody remains lightweight — it only tracks what matters. The helper scripts are equally essential: without them, every consumer would need to understand the metadata file format, defeating the purpose of the abstraction.

**Independent Test**: Generate views, create a mental model for a view, then modify a source file and commit. Run the staleness checker. Verify the mental model sections referencing the changed file are flagged invalid, and that the root mental model sections referencing that view mental model are also flagged invalid.

**Cucumber Test Tag**: `@view-mental-models`

**Acceptance Scenarios**:

1. **Given** a generated view, **When** the view-agent produces a mental model, **Then** it creates the mental model in `mental-models/`, an immutable metadata file containing diff entries keyed by section path (each with curated input file paths and content hashes), and eagerly copies the selected input files to a snapshot directory alongside the metadata file.
2. **Given** a views root, **When** the root agent produces a root mental model, **Then** it creates a root-level `views/mental-models/` directory with a metadata file whose diff entries reference child view mental-model sections (not individual source files — the root's inputs are the child mental models), eagerly copied to the snapshot directory.
3. **Given** an existing mental model, **When** a view-agent updates it, **Then** a new immutable metadata file is created (the previous one is never modified). Each diff entry in the new metadata file points back to the diff entry in the previous metadata file that it supersedes. New input files are eagerly copied to a snapshot directory. The head metadata file references live repository paths.
4. **Given** the head metadata file, **When** any of the selected source files' content hash differs from the recorded hash (committed or uncommitted changes), **Then** the rendering script flags the diff entries (and their corresponding mental model sections) referencing that file as stale/invalid.
5. **Given** a view-level mental model flagged as stale, **When** the root-level mental model rendering script runs, **Then** the root diff entries that reference the stale view mental model are also flagged as stale (cascading invalidation).
6. **Given** a mental model with mixed valid and stale diff entries, **When** the rendering script runs, **Then** it produces output that clearly distinguishes current sections from stale sections.
7. **Given** a stale diff entry, **When** the staleness is caused by a small diff to a selected source file, **Then** the system MAY resolve it by creating a new metadata file with updated line references without requiring a full mental model regeneration.
8. **Given** any metadata file, **When** an agent follows the previous-metadata pointers in each diff entry, **Then** it can walk backwards through the full history of that mental model section, with all input files available as local snapshots at each step — no git operations needed.
9. **Given** a mental model, **When** the LLM runs `view-model show`, **Then** it receives the rendered mental model content with staleness annotations and the list of curated references for that model — no metadata file paths or internal structure are exposed.
10. **Given** a mental model, **When** the LLM runs `view-model files` for a specific section, **Then** it receives the list of curated references for that section, with current/stale status for each.
11. **Given** updated content and a list of section references, **When** the LLM runs `view-model update`, **Then** the script creates the new immutable metadata file, copies snapshots, links diff entries, and updates the head pointer — the LLM does not interact with any metadata files.
12. **Given** a mental model section, **When** the LLM runs `view-model add-ref` with a new source file path or child view section reference, **Then** the reference is added to that section's curated set, a new metadata file is created, and the snapshot is copied — without changing the mental model content itself.
13. **Given** a view whose `regen.py` script has changed since the last metadata file was created, **When** the staleness check runs, **Then** ALL sections in that view's mental model are flagged for review — the agent must determine whether the view's file set has changed and whether the mental model sections are still valid. Additionally, the staleness message identifies which specific view's script changed and references the script diff so the agent can inspect the change.
14. **Given** a view whose `regen.py` has changed, **When** the root mental model staleness check runs, **Then** only the root diff entries referencing that specific view's mental models are flagged — other views' root diff entries remain unaffected (granular root invalidation).

### User Story 3 — Docker-Scoped View-Agent CLI (Priority: P1)

A standalone Python CLI tool (`view-agent`) accepts a query/task, a view path, and an Ollama model name. Each invocation operates on a **single target view path** inside a Docker container. The repository root is mounted read-only at a stable path so relative view symlinks resolve and git hashes can be computed. Only the target view's `mental-models/` directory (or `views/mental-models/` in root mode) is mounted read-write.

Every view-agent invocation follows a two-step process:

1. **Staleness check & refresh**: The agent reads the head metadata file. In view mode, it compares the recorded content hashes against the current hashes of the curated source files, and also compares the recorded view script hash against the current hash of the view's `regen.py`. If the view script has changed, all sections in that view are flagged for review (the view's file composition may have changed). In root mode, there is no root `regen.py`; instead the agent compares child head metadata IDs and referenced child section statuses. For any stale sections, the agent researches the changed inputs, updates the affected mental-model sections, and creates a new immutable metadata file with updated pointers. This ensures the mental model is current before answering any query.

2. **Query processing**: The agent loads the (now-current) mental model and view files, queries the host Ollama instance, and outputs a structured response. If the query results in further mental model insights, those are also written back, producing another metadata file.

The CLI is designed to be invoked N times in parallel by the discovery agent (one `docker run` per view), plus one additional invocation for the root mental model (with the entire `views/` directory mounted). This Docker scoping prevents writes outside the target mental-model directory while still giving the container the repository read access required for symlink resolution and staleness checks.

**Why this priority**: This is the execution engine — it is the mechanism by which the discovery agent fans out work across views using local models. Docker sandboxing is essential because agents modify mental models, and without isolation an agent could corrupt another view's state. The built-in staleness check means mental models are always refreshed on use — there is no separate update process.

**Independent Test**: Create two small test views with a few files each. Modify a source file in one view. Run two `docker run` commands in parallel, each mounting the repository read-only and the target view's `mental-models/` directory read-write. Verify that the agent for the modified view detects the stale section, refreshes the mental model, and then answers the query with current context.

**Cucumber Test Tag**: `@view-agent-cli`

**Acceptance Scenarios**:

1. **Given** a view directory and a query, **When** the CLI is invoked via `docker run` with the repository mounted read-only and the target mental-model directory mounted read-write, **Then** the agent returns a structured response and persists updates only under that target mental-model directory.
2. **Given** a view with a mental model whose head metadata file shows all sections current, **When** the CLI agent starts, **Then** it skips the refresh step and proceeds directly to query processing.
3. **Given** a view with a mental model whose head metadata file shows stale sections (source file hashes have advanced), **When** the CLI agent starts, **Then** it first researches the changed files, updates the stale mental model sections, creates a new immutable metadata file with refreshed pointers and snapshots, and only then proceeds to query processing with the now-current mental model.
4. **Given** a view with a mental model, **When** the CLI agent processes the view, **Then** it includes the mental model content in the context sent to the Ollama model and can write updates back to the mental model through the mounted volume.
5. **Given** mental model updates written by the agent, **When** the container exits, **Then** the updates persist on the host filesystem, a new immutable metadata file is created with diff entries linking back to the previous metadata file, and the curated input files are eagerly copied to the new snapshot directory.
6. **Given** an Ollama model that is not available on the host, **When** the CLI is invoked, **Then** it reports a clear error identifying the missing model and exits gracefully.
7. **Given** a view-agent container, **When** it attempts to write outside its target mental-model directory, **Then** the write fails because only that directory is mounted read-write.

### User Story 4 — Discovery Agent Integration (Skills + Prompt Driven) (Priority: P2)

The discovery agent integration is delivered by loading the view-agent skills and updating the discovery prompt. When the configured discovery agent chooses to use those skills, it follows a two-phase flow to parallelize research across views. For automated integration testing, the configured prompt/profile must explicitly prefer this path whenever generated views are present.

**Phase 1 — Per-View Fan-Out**: The discovery agent detects available views, then launches one `docker run` command per view in parallel. Each container targets one view path under the read-only repository mount, with write access only to that view's `mental-models/` directory. All containers run concurrently against the host Ollama instance. The discovery agent waits for all containers to return their per-view responses.

**Phase 2 — Root Synthesis**: The discovery agent launches one more `docker run` command for the root view-agent, this time mounting the entire `views/` directory (including the now-updated per-view mental models). This root agent synthesizes the per-view responses into the root mental model and returns a consolidated answer. The discovery agent then performs any additional targeted research and returns.

**Why this priority**: This is the integration point that makes the system useful end-to-end, but it depends on Stories 1, 2, and 3 being complete.

**Independent Test**: Configure the discovery agent with the view-agent skills loaded and an integration-test prompt/profile that explicitly prefers the view-agent workflow whenever generated views are present. Against a repository that has generated views, issue a discovery request. Verify that the agent launches parallel Docker containers (one per view), waits for all to complete, then launches a root container, and incorporates the final result into its response.

**Cucumber Test Tag**: `@view-agent-discovery-integration`

**Acceptance Scenarios**:

1. **Given** a repository with generated views and a discovery agent configured with the view-agent skills and integration-test prompt guidance that prefers the view-agent workflow, **When** the discovery agent receives a research request, **Then** it launches one Docker container per view in parallel, each targeting a single view path.
2. **Given** all per-view containers have completed, **When** the discovery agent proceeds to Phase 2, **Then** it launches a root container with the entire `views/` directory mounted, which synthesizes results into the root mental model.
3. **Given** the root container has completed, **When** the discovery agent receives the consolidated response, **Then** it incorporates the result into its final answer, referencing specific views where relevant.
4. **Given** a repository with no views, **When** the discovery agent receives a request, **Then** it falls back to its normal (non-parallelized) behavior.
5. **Given** a per-view container that times out, **When** the discovery agent detects the timeout, **Then** it proceeds with the responses from the containers that completed, noting which views timed out.

### User Story 5 — Chain of Custody History & Search (Priority: P2)

The chain of custody is not just a point-in-time staleness check — it is a **historical record** of how mental models have evolved alongside the curated inputs that produced them. The history is captured structurally through the immutable metadata files (Story 2): each metadata file's diff entries point back to the previous metadata file they supersede, and each carries eagerly-copied snapshots of those inputs at that point in time. This forms a flat collection of immutable files — like a content-addressed hash map — where walking backward is pointer-following, not log parsing.

A **search/query tool** provides the ability to traverse and reason over this structure. An agent or developer can: walk backwards through the chain for a specific mental model section (seeing each prior version alongside the exact inputs that produced it), query which metadata files reference a particular source file or child view section, filter by date, and for view-level chains filter by source-file content-hash range, and see how the curated set of important inputs evolved over time — all without performing any git operations, because those inputs are saved as snapshots in each metadata file's snapshot directory.

The system provides:

1. **A search/query CLI tool** — operates over the flat collection of immutable metadata files per view. It can traverse backward pointers, filter by source file path or child view reference, mental model section path, date range, or source-file content-hash range for view-level chains. The tool reads the metadata files and their snapshot directories directly — no database or index needed, since the structure is already a flat map of immutable files.

This serves two purposes: (1) agents can reason about the *current* mental models by understanding their provenance (which files or child sections, which content hashes, which prior versions), and (2) agents and developers can see how mental models have evolved over time and what caused each revision, including how the curated input selection itself changed.

**Why this priority**: P2 because the basic chain of custody (Story 2) provides the immutable metadata structure, but without search, agents can't efficiently reason over the history. The search tool makes the linked-list structure queryable.

**Independent Test**: Generate views, create a mental model, update it twice (after two source file changes). Use the search tool to walk backwards from the head metadata file for a specific section — verify it shows three metadata files (initial + two updates) each with their snapshot directories.

**Cucumber Test Tag**: `@view-custody-history`

**Acceptance Scenarios**:

1. **Given** a chain of three immutable metadata files (initial, update-1, update-2), **When** an agent uses the search tool to walk backwards from the head for a specific mental model section, **Then** it sees each prior diff entry alongside the exact input file snapshots that produced it — by following pointers, not parsing a log.
2. **Given** a collection of metadata files, **When** a user or agent queries by a specific source file path, **Then** all metadata files whose diff entries reference that file (in their curated input set) are returned, showing how the file's role in the mental model evolved over time.
3. **Given** a collection of view-level metadata files, **When** a user or agent queries filtered by a source-file content-hash range, **Then** the matching metadata files are returned.
4. **Given** a collection of metadata files, **When** a user or agent queries by mental model section path, **Then** the chain of metadata files contributing to that section is returned in reverse chronological order.

### Edge Cases

- What happens when a repository has thousands of files — does view generation complete in a reasonable time?
- What happens when symbolic links point to files that have been moved or renamed?
- What happens when the Ollama service is not running when the CLI is invoked inside a Docker container?
- What happens when a single view-agent container takes significantly longer than others — does the discovery agent handle timeouts?
- What happens when the views directory already exists from a previous run — is it safely overwritten or merged?
- What happens when two view-agents produce contradictory answers during root synthesis?
- What happens when a mental model's metadata chain references a file that has been deleted from the repository?
- What happens when a view-agent container crashes — are partial mental model updates left in a corrupt state, or is there atomic write behavior?
- What happens when the Docker daemon is not running — does the discovery agent detect this and fall back gracefully?
- What happens when every section of a mental model is stale — is the entire model treated as needing regeneration?
- What happens when the collection of immutable metadata files and snapshot directories grows very large on disk — is there a pruning or archival strategy for old metadata files?
- What happens when the search tool traverses a very long chain of metadata files — does it remain performant?
- What happens when a file previously in the curated set is removed by the LLM in a subsequent update — the new metadata file simply won't reference it, and the old one remains immutable showing it was once included.
- What happens when snapshot directories contain large files — is there deduplication across metadata files when the same file content appears in consecutive snapshots?
- What happens when a view's `regen.py` script changes but the resulting symlink set is unchanged — is there a way to detect this and skip full regeneration? (Initial approach: flag all sections for review and let the agent decide; optimization deferred.)
- What happens when a ticket adds new files that should belong to an existing view but the view's `regen.py` doesn't include them — the agent must re-examine and potentially update that view's `regen.py` script.
- What happens when one view's `regen.py` changes but other views' scripts are unchanged — only the changed view's mental models and the root sections referencing that view are flagged; other views are unaffected (granular invalidation).

## Requirements *(mandatory)*

### Functional Requirements

#### View Generation

- **FR-001**: The system MUST analyze a repository and produce one or more named views, each containing **relative** symbolic links to a coherent subset of repository files. Symlinks MUST use relative paths (via `os.path.relpath`) so views work both on the host filesystem and inside Docker containers where the repo is mounted at a different path.
- **FR-002**: Views MUST be stored under a `views/` directory at the repository root (or a configurable location).
- **FR-003**: Each view MUST include its own `regen.py` Python script (agent-generated, project-specific) that can recreate that view's symbolic links from the current repository state. Each script is tailored to its specific view's file grouping. Having one script per view (rather than one global script) enables granular staleness: when a view's script changes, only that view and its root references are flagged.
- **FR-004**: The view generator MUST support a tree-based traversal strategy that groups files by directory structure, producing hierarchical views (e.g., per-module, per-package, per-feature-area).
- **FR-005**: The view generator MUST produce a fallback "all-files" view when no meaningful grouping can be determined.

#### Mental Models & Chain of Custody

- **FR-006**: Each view MUST have a `mental-models/` subdirectory containing exactly one mental model document (`mental-models.md`). One view = one mental model = one HEAD = one metadata chain.
- **FR-007**: A root-level `views/mental-models/` directory MUST exist containing exactly one root mental model document (`mental-models.md`) for cross-view knowledge.
- **FR-008**: Mental models MUST be plain markdown files. Each view has exactly one (`mental-models.md`), with sections organized as markdown headings within that file.
- **FR-009**: Each mental model MUST be accompanied by an **immutable metadata file** — a structured file containing diff entries, where each diff entry maps a mental-model section hunk (identified by unique section path) to the curated inputs selected as evidence and a pointer to the previous metadata file's diff entry that this one supersedes. For view-level metadata, curated inputs are source files and the metadata file MUST also record the content hash of that view's `regen.py` script at creation time. For root-level metadata, curated inputs are child mental-model references and no root script hash is recorded because the root has no `regen.py`.
- **FR-009a**: Each diff entry in the metadata file MUST record: the section path it covers (unique heading hierarchy, e.g., `Architecture Overview > Data Flow`), the section offset (ordinal position — display metadata only), and exactly one curated input set: either source file paths plus their content hashes (view-level) or child view references plus child section paths/head metadata IDs (root-level). For non-initial entries it MUST also record a pointer to the previous metadata file and diff entry it supersedes (via section path).
- **FR-009a2**: Each metadata file MUST have an associated **snapshot directory** containing eagerly-copied versions of the curated input files at the time of production. The head metadata file's diff entries MUST also reference the live repository paths.
- **FR-009b**: For view-level mental models, a rendering script MUST compare the head metadata file's recorded content hashes against current working-tree content hashes (`git hash-object`) and flag diff entries where any contributing file's hash differs as **stale/invalid**. This catches both committed and uncommitted changes. Only files in the curated set are checked. Additionally, for view-level mental models the rendering script MUST compare the recorded view script hash against the current hash of the view's `regen.py` — if the script has changed, ALL diff entries for that view are flagged for review.
- **FR-009c**: Root-level mental model metadata files MUST have diff entries that use **ChildModelRefs** (not SourceFileRefs) — referencing child views by view name, child section path, and child head metadata ID, not by file path and content hash. A root diff entry is stale when its child view's head metadata ID has changed or any of its referenced child sections are stale. When a specific view's `regen.py` changes, only the root diff entries referencing that view are flagged — other views' root diff entries are unaffected.
- **FR-009d**: The rendering script MUST produce output that clearly distinguishes current (valid) sections from stale (invalid) sections.
- **FR-009d2**: When staleness is caused by a small diff to a selected source file, the system SHOULD support lightweight resolution (creating a new metadata file with updated line references) without requiring full mental model regeneration.

#### Mental Model Helper Scripts

- **FR-009-show**: A `view-model show` command MUST render the current mental model with staleness annotations (current/stale per section) and the list of curated references for each section. For view-level models, those references are source files. For the root model, those references are child view mental-model sections. No metadata file paths or internal structure may be exposed in the output.
- **FR-009-files**: A `view-model files` command MUST list the curated references for the mental model (or a specific section), with current/stale status for each reference. For view-level models this returns source file references; for the root model this returns child mental-model references.
- **FR-009-update**: A `view-model update` command MUST accept updated mental model content and a list of section references keyed by section path. Each section reference MUST contain either source file references (paths + optional source spans using `start_line` + `line_count`) or child model references (view name + child section paths), depending on the mental model scope. The command MUST handle all metadata mechanics internally: creating the new immutable metadata file, copying snapshots, linking diff entries to the previous metadata file, updating the head pointer. The caller provides only content and references.
- **FR-009-addref**: A `view-model add-ref` command MUST add a curated reference to an existing mental model section without changing the mental model content, creating a new metadata file with the expanded curated set. The added reference is a source file for view-level models or a child mental-model reference for the root model.
- **FR-009-status**: A `view-model status` command MUST report which sections are current vs. stale without rendering the full mental model content.
- **FR-009-addsection**: A `view-model add-section` command MUST add a new section to the mental model. The command MUST validate that the section's heading path is unique within the document. Section paths are the full heading hierarchy joined by ` > ` (e.g., `Architecture Overview > Data Flow`). Duplicate section paths MUST be rejected with a clear error. This is the only way sections should be added — it enforces the uniqueness constraint that the metadata chain depends on.
- **FR-009-abstract**: The LLM and all other consumers (view-agent CLI, search tool, developers) MUST interact with mental models exclusively through the helper scripts. No consumer should need to read or write metadata files directly. The mental model file itself should only be modified through `view-model update` or `view-model add-section`.

#### Chain of Custody History & Search

- **FR-009e**: The immutable metadata files and their snapshot directories MUST be stored as a flat collection in the view's `mental-models/` directory. Each metadata file is identified by a unique name (e.g., timestamp-based or content-hash-based). No metadata file is ever modified after creation.
- **FR-009f**: The helper scripts (FR-009-show through FR-009-status) MUST be the reusable implementation of the mechanical operations for creating metadata files, copying input file snapshots, and linking diff entries to previous metadata files, since the same structure is used for both view-level metadata (input = source files) and root-level metadata (input = child view mental-model sections).
- **FR-009g**: A search/query CLI tool MUST support traversing the metadata file chain by: walking backwards from the head following "supersedes" pointers, filtering by source file path (view-level), child view name (root-level), child section path within a child view (root-level), mental model section path, date range, and source-file content-hash range (view-level).
- **FR-009g2**: The search tool MUST support walking backwards through the history of a specific mental model section, resolving the snapshot directory at each step to show the exact inputs that produced that version — without any git operations.
- **FR-009k**: The view-agent CLI MUST create a new immutable metadata file (with snapshot directory) whenever it creates or modifies a mental model section.

#### Sub-Agent CLI (Docker-Scoped)

- **FR-010**: The CLI MUST be a standalone Python tool packaged in a Docker image, invokable via `docker run`.
- **FR-011**: The CLI MUST accept a query, a single target view path inside the container, and an Ollama model name as inputs.
- **FR-012**: Each CLI invocation operates on exactly one view. Parallelism is achieved by the caller (discovery agent) launching multiple `docker run` commands concurrently.
- **FR-013**: The agent MUST load the target view's file listing and mental model content, construct a prompt, and send it to the host's Ollama instance (accessed via `host.docker.internal` or equivalent).
- **FR-014**: The CLI MUST output its response as structured JSON to stdout.
- **FR-015**: The CLI MUST support a `--mode root` flag for the root synthesis invocation, where the entire `views/` directory is mounted and the agent synthesizes per-view mental models into the root mental model.
- **FR-015a**: As the first step of every invocation, the CLI MUST check the head metadata file for stale sections. In view mode, it does this by comparing recorded content hashes against current working-tree content hashes of curated source files AND comparing the recorded view script hash against the current hash of the view's `regen.py`. If the view script has changed, all sections in that view are flagged for review. In root mode, it does this by comparing recorded child head metadata IDs and referenced child section statuses; the root has no `regen.py`. If stale sections are found, the CLI MUST research the changed inputs (comparing the snapshot copy against the live file or child model to produce a diff for the LLM), update the affected mental model sections, and create a new immutable metadata file before proceeding to query processing.
- **FR-016**: The CLI MUST handle Ollama connection errors gracefully with clear error messages.
- **FR-017**: The CLI MUST support a configurable timeout to prevent indefinite hangs.
- **FR-017a**: The CLI MUST create a new immutable metadata file (with eagerly-copied input file snapshots) for any mental model sections it creates or modifies, linking diff entries back to the previous metadata file.
- **FR-017b**: The Docker images MUST be built from Dockerfiles in the respective Python package directories (`view_agents_utils/docker/` and `view_agent_exec/docker/`) so they can be versioned and rebuilt as the CLI evolves.
- **FR-017c**: `view-model update`, `view-model add-ref`, `view-model add-section`, and any view-agent refresh flow MUST update the mental model atomically. On failure, the previous markdown file and `HEAD` pointer MUST remain intact. The implementation MUST also serialize updates per mental model so concurrent refreshes do not corrupt the chain.

### Key Entities

- **View**: A named directory of symbolic links into a repository, representing a coherent subset of the codebase. Contains its own `regen.py` Python script (agent-generated, specific to this view) and a `mental-models/` subdirectory. Each view's `regen.py` hash is tracked independently in that view's metadata files for staleness detection — a script change in one view does not affect other views.
- **Mental Model**: A single markdown document (`mental-models.md`) per view describing the architecture, patterns, and key decisions of code within that view. Each view has exactly one mental model, one HEAD symlink, and one metadata chain. The root also has exactly one mental model for cross-view synthesis.
- **Metadata File**: An immutable structured file containing diff entries and a scope. Each diff entry is keyed by section path and records: SourceFileRefs (view-level: repo path + working-tree content hash) or ChildModelRefs (root-level: view name + child section path + child head metadata ID), a pointer to the previous metadata file's diff entry it supersedes (via section path), and a reference to the associated snapshot directory. View-level metadata files also record the content hash of the view's `regen.py`; root-level metadata files do not because the root has no script. Never modified after creation.
- **View Agent**: A single Docker container running the CLI tool against one target view path — loads files and mental model, queries the host Ollama instance, returns a response, and updates only that target mental-model directory.
- **Root Agent**: A special view-agent invocation (`--mode root`) with the entire `views/` directory mounted, responsible for synthesizing per-view mental models into the root mental model. The root has no `regen.py`; it tracks custody through child view mental-model references.
- **Snapshot Directory**: A directory associated with each metadata file containing eagerly-copied versions of the curated input files at the time of that metadata file's creation. Makes history self-contained — no git needed to see past states.
- **Chain of Custody**: The linked chain of immutable metadata files from head to initial, connected by "supersedes" pointers in diff entries (linked by section path). Forms a flat hash-map-like structure. When a source file's content hash differs from the recorded snapshot, the head metadata file's diff entries referencing it are flagged stale. At the root level, entries are stale when their referenced child view sections are stale or their child view's head metadata ID has changed. When a view's `regen.py` hash changes, all diff entries in that view's mental models are flagged for review, and only the root diff entries referencing that specific view cascade — other views remain unaffected.
- *Custody Visualization*: Deferred to a future iteration. Will provide static HTML/SVG files showing the chain of custody as a graph.

## Success Criteria *(mandatory)*

- **SC-001**: View generation completes for a repository of up to 5,000 files in under 60 seconds.

- **SC-009**: Cascading invalidation from source file → view mental model → root mental model is detected in a single rendering pass (no iterative convergence needed).
- **SC-010**: Search tool queries return results in under 2 seconds when traversing a chain of up to 100 metadata files.
- **SC-011**: An agent can use the search tool to determine the full provenance of any mental model section — which source files or child view sections, which prior versions, and the input snapshots at each step — in a single CLI call.

## Assumptions

- Ollama is installed and running locally on the developer's machine, with at least one model pulled. The Docker containers access the host Ollama via `host.docker.internal` (macOS/Windows) or `--network host` (Linux).
- Docker is installed and the Docker daemon is running on the developer's machine.
- The repository being viewed is a local git repository (not a bare or remote-only repo).
- Symbolic links are supported by the filesystem (standard on macOS and Linux). View symlinks MUST be relative. In Docker, the repository root is mounted read-only at `/repo`, and the target view is passed as a subpath under that mount so those relative symlinks resolve correctly.
- The Python CLI targets Python 3.11+ and runs inside Docker containers.
- View generation uses agent-generated, per-view Python scripts (not a generic tool) — each view directory contains its own `regen.py`. The initial strategy is directory-structure-based grouping; more sophisticated strategies (semantic, dependency-graph) are deferred to later iterations.
- Two Python packages are scaffolded from `multi_agent_ide_python_parent/packages/app_template`: `view_agents_utils` (metadata, custody, helpers) and `view_agent_exec` (Ollama query execution). Each produces its own Docker image.
- Three skills provide the interface layer: `multi_agent_ide_view_generation` (instructional only — documents the view concept and how to generate Python scripts), `multi_agent_ide_view_agents` (Python scripts invoking the `view-agents-utils` container), and `multi_agent_ide_view_agent_exec` (Python scripts invoking the `view-agent-exec` container).
- Discovery agent integration (Story 4) will be implemented by loading the relevant skills and updating the discovery prompt in `multi_agent_ide_view_agent_exec`, rather than by introducing a new hard-coded Java orchestration path.
- Git must be available inside the Docker containers for hash resolution. Both Docker images include git.
